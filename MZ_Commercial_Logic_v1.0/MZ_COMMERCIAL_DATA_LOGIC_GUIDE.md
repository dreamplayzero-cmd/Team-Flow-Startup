# MZ 상권분석 AI: 데이터 및 스코어링 로직 가이드 (Developer & User Guide)

본 문서는 **MZ 상권분석 AI 프로젝트**에서 수집되는 4가지 이기종 데이터(인구, 상가, 텍스트 감성, 검색 트렌드)의 구조체, 수집 현황, 그리고 이 데이터를 활용해 성공 확률을 도출하는 득점(Scoring) 알고리즘을 상세히 설명하기 위해 작성되었습니다.

## 🚀 [NEW] Dual Sync & Override 아키텍처 (2026-04-17)

MZ 상권분석 앱은 데이터 단절과 속도 제한(Rate Limit)을 영리하게 극복하기 위해 두 가지 데이터 로드 전략을 병행해서 사용합니다.

1. **디폴트 모드 (Automated API + Fallback)**
   - 관리자가 지역을 추가하고 동기화(Sync) 버튼을 누르면 인구, 네이버 검색, 블로그 감성 정보가 실시간 네트워크를 타고 DB로 꽂힙니다.
   - 이때 상가 API가 막혀있더라도 엔진이 깡통으로 멈추지 않게 **결정론적 Fallback Mock 데이터(약 50점 부여)**를 1초 만에 자동 생성해 넣습니다.
2. **이벤트 모드 (CSV Manual Override)**
   - 깡통(Fallback) 상가 데이터 대신 무시무시한 정밀도를 가진 **초정밀 상권 분석**을 하고 싶을 때 관리자가 발동하는 모드입니다.
   - 소상공인진흥공단에서 다운받은 대용량 상가 CSV 파일(290MB)을 관리자가 업로드하면, `CsvImportWorker(QThread)`가 깨어나 타겟 지역 상가만 필터링한 뒤 빈약했던 기존 상가 데이터를 강력하게 **덮어씌웁니다(Override).**
   - 위 두 가지 로직이 완전히 분리되어 앱은 멈추지 않고, 최신의 통계를 반영할 수 있는 완전체(Dual) 시스템입니다.

### 👥 1) 인구 데이터 (population_data)
행정안전부(admmSexdAgePpltn) API를 취합하여 해당 지역에 거주하는 대상(MZ세대 중심)의 인구 밀집도를 파악합니다.
- **현재 수집량**: `9 건`
- **저장 항목**:
  - `area_code` (지역코드), `age_group` (연령대: MZ), `gender` (성별), `population_count` (인구수), `reference_date` (통계 기준일)

### 🏪 2) 상가 정보 데이터 (store_info)
소상공인시장진흥공단(ODCloud) 데이터를 기반으로 특정 상권 내 위치한 경쟁 업체들의 분포와 개수를 파악합니다.
- **현재 수집량**: `120 건`
- **저장 항목**:
  - `store_id` (고유번호), `area_code` (지역코드), `store_name` (상호명), `category_large/medium/small` (업종 분류), `latitude/longitude` (위경도)

### 📈 3) 검색어 트렌드 분석 (keyword_trends)
네이버 검색어 API(DataLab)를 활용하여 특정 상권+업종(예: "성수동 카페")에 대한 온라인 수요(관심도)를 파악합니다.
- **현재 수집량**: `4 건` (API 레이트 리밋에 맞춰 점진적 수집)
- **저장 항목**:
  - `area_code` (지역코드), `keyword` (검색어), `search_volume` (검색 볼륨 지수), `search_platform` (플랫폼, 예: Naver), `target_month` (조회 월)

### 💬 4) 블로그 여론 및 감성 분석 (blog_analysis)
네이버 블로그 크롤링을 통해 해당 상권의 트렌디함과 실제 방문객들의 여론(긍정/부정)을 분석합니다.
- **현재 수집량**: `43 건`
- **저장 항목**:
  - `area_code` (지역코드), `keyword` (분석 키워드), `mention_count` (언급량), `sentiment_score` (감성 점수 0.0~1.0), `analysis_date` (분석 일자)

## 🧮 2. 개별 데이터의 점수화(Scoring) 로직 계산식

앱 내의 `ScoringEngine` 모듈(`core/engine/scoring_engine.py`)은 4가지 특징이 전혀 다른 데이터를 **100점 만점의 규격화된 점수**로 변환합니다.

### A. 인구 점수 (유동/거주 수요 밀집도)
* **목표**: 상권에 잠재 고객(MZ세대)이 얼마나 밀집해 있는가?
* **계산 로직**: `min((해당 지역 MZ 인구 총합 / 10,000명) * 100, 100)`
* **설명**: 1만 명을 기준점(100점 만점)으로 삼고 비율로 계산해 폭발적인 잠재 수요가 존재하는지 확인합니다. (0점에 가까울수록 유령 상권)

### B. 수요 점수 (온라인 검색 관심도)
* **목표**: 사람들이 이 지역의 특정 업종을 얼마나 능동적으로 검색하는가?
* **계산 로직**: `min(최대 검색 볼륨 지수, 100)`
* **설명**: 포털 검색량은 방문 의지를 대변하는 최고 지표입니다. 수치가 높을수록 뚜렷한 활성화가 이루어진 검증된 시장을 의미합니다.

### C. 트렌드 점수 (블로그 감성 여론)
* **목표**: 상권 긍정도 및 소셜 바이럴 효과가 뛰어난가?
* **계산 로직**: `블로그 평균 감성 점수(0.0~1.0) * 100`
* **설명**: 맛, 분위기, 핫플 등의 긍정 키워드가 추출될 때 1.0에 가까워집니다. 이를 100점 만점으로 스케일링합니다.

### D. 경쟁 점수 (경쟁 강도 반비례)
* **목표**: 내가 들어갈 자리가 있는가? (경쟁자의 수가 중요)
* **계산 로직**: `( 1 / (동일 업종 상가 수 + 1) ) * 100`
* **설명**: 경쟁 상가 개수가 역수로 작동합니다. 기존 업체가 0개면 무주공산(100점)이나, 경쟁업체가 많아질수록 기하급수적으로 점수는 하락하여 레드오션을 의미하게 됩니다.

## 🧠 3. 최종 통합 점수 산출 (Business Integration & Dynamic Logic)

### 📊 동적 가중치(Dynamic Weights) 시스템
기존의 고정형(각 25%) 합산 방식에서 벗어나, 사용자가 입력한 **'매장 운영 형태'**에 맞추어 4대 데이터의 비중이 역동적으로 변환됩니다.

- **기본형**: 인구(25%) + 검색(25%) + 매장(25%) + 트렌드(25%) 
- **배달 중심**: 유동인구 📉(10%) / 검색 수요 📈(35%) / 트렌드 📉(20%) / 경쟁 매장 폭 📈(35%)
- **테이크아웃 중점**: 유동인구 📈(40%) / 검색 수요 📉(15%) / 트렌드 📈(30%) / 경쟁 매장 폭 📉(15%)

### ⚔️ 4대 스마트 페널티 & 버프 제어 장치
최종 계산된 점수를 기반으로 창업자의 조건(나이, 경력, 자본 등)에 맞춘 **맞춤형 페널티/버프**를 더해 가장 현실적인 성공률(Probability)을 산출합니다.

1. **[버프] 베테랑 생존 방어 (경력 3년 이상)**: +5점 매칭 (잔뼈가 굵은 노하우 보정)
2. **[버프] 자본재 마케팅 버퍼 (자본금 1억 이상)**: +3점 매칭 (버티기 체력 및 초반 홍보력 증강)
3. **[페널티] 생초보 진입 불가 레드오션 (무경력 + 경쟁지수 최악)**: -10점 강제 차감 (폐업률 99% 억제용 안전장치)
4. **[페널티] 예산 파산 리스크 (프리미엄 상권 + 5천만원 이하 예산 + 홀형 매장)**: -8점 차감 (임대료 및 고정비 초과 위험 경고)
5. **[특수] 심야 핫플 궁합 (심야 영업 + 이태원·한남동)**: +5점 추가 (유흥가 버프) / 주거/대학상권: -5점 차감
6. **[경고] 유령 젠트리피케이션 장벽 (트렌드 & 검색 지수 복합 하락장)**: -10점 강력 경고

### 📝 다이내믹 인사이트 리포트 생성 로직
엔진은 산출된 점수에 맞춰 숫자를 제공함과 동시에 **사용자 맞춤형 인사이트(Pros/Cons)** 컨설팅을 제공합니다.
각 섹션이 특정 점수(`> 70점` 또는 `< 30점`) 돌파 시 또는 특정 페널티가 활성화될 경우 텍스트를 실시간으로 조립하여 앱 화학 결과 화면에 표출합니다. (예: *"경력 부족 및 레드오션: 초보자가 살아남기 힘든 초경쟁 상태입니다"* 등)
