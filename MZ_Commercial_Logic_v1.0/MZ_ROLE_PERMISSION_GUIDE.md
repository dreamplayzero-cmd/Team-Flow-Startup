# 🔐 MZ 상권분석 AI 애플리케이션: 권한 계층(Role Permission) 아키텍처 가이드

본 문서는 프로젝트의 보안 및 권한 분리를 명확히 하기 위해 **일반 창업자(User)**와 **시스템 구동 관리자(Admin)** 간의 접근 가능 영역 및 세부 기능 제어 현황을 정리한 문서입니다.

---

## 👥 1. 일반 사용자 (Entrepreneur / User) 권한
**대상**: 실제 창업을 준비하며, 다양한 데이터를 바탕으로 최적의 상권을 추천 및 예측 받고 싶은 소상공인.

### 🔓 접근 가능 영역 (Front-End)
일반 사용자는 오직 **"데이터 입력"**과 **"결과 확인"**만 수행할 수 있으며 데이터베이스 변경이나 시스템 수집 제어 권한이 원천 차단됩니다.

* **[화면 1] 로그인 화면 뷰 (`LoginView`)**
  * 일반 창업자 전용 버튼을 통해 패스워드 없이 즉시 `UserDashboard`로 진입 가능.
* **[화면 2] 조건 입력 대시보드 (`InputView`)**
  * **보유 자본금 및 투자 예산** 설정 (슬라이더 / 콤보박스)
  * **목표 상권 선택** (다중 지역 체크박스, 최대 4개 지역 동시 분석)
  * **프랜차이즈 vs 개인 매장** 여부 선택
  * **매장 운영 형태** (홀 테이블, 배달 전문, 테이크아웃 등 가중치 반영 옵션)
  * 본인의 창업 및 동일 업계 **경력 년수** 입력
* **[화면 3] 분석 결과 시각화 모니터 (`ResultView`)**
  * AI가 도출한 100점 만점 기준의 상권별 최종 매칭률 / 생존율 확인
  * 가장 유리한 생태계(🏆 베스트 상권) 조회
  * 점수 하락 요인(Risk) 및 상승 요인(Opportunity) 시뮬레이션 페널티 리포트 열람.
  * *권한 제한*: 결과값 출력 이면의 4대 RAW 데이터(원천 로우 데이터)나 API 접근은 절대 불가능.

## 🛡️ 2. 시스템 관리자 (System Administrator) 권한
**대상**: 애플리케이션의 유지 보수, 데이터 업데이트, 비즈니스 알고리즘 매개변수를 제어하는 최고 권한자(개발자 또는 운영 매니저).

### 🔑 접근 가능 영역 (Back-End Control)
관리자는 앱의 시작화면에서 **[관리자 전용 로그인]** 채널을 이용해 진입하며, 시스템의 "심장과 뇌"를 조정할 수 있습니다. 데이터를 추가, 덮어쓰기하거나 시스템 리소스를 강제 할당할 수 있습니다.

* **[기능 1] 백그라운드 4대 데이터 즉시 수집(Auto Sync) 제어**
  * 네이버 블로그/검색 트렌드, 행정안전부 인구데이터, 상가 초기 데이터를 실시간으로 강제 크롤링 및 API 갱신하도록 명령할 수 있습니다.
  * UI 응답 없음(Freezing) 방지를 이끄는 `DataSyncWorker` 스레드를 직접 On/Off 발동시킵니다.
* **[기능 2] 대용량 기밀/정밀 데이터 직접 주입 (CSV Override)**
  * **권한 독점**: 일반 유저는 접근이 불가한 `AdminView` 탭을 통해서만 소상공인시장진흥공단의 보안 CSV(300MB 대용량) 물리 파일을 데이터베이스에 덮어쓸 수 있습니다.
  * **단절 모드 통제**: API 가동이 막혔을 때 앱이 무사히 돌아가도록 오프라인 파일을 삽입해 앱을 소생시키는 핵심 마스터키를 소유합니다.
* **[기능 3] 시스템 상태 모니터링 (`Status Log`)**
  * 수집 중 오류, 실패 내역, 진행도(0~100%) 시스템 메시지를 실시간으로 추적 및 로그화하여 볼 수 있습니다.

## ⚙️ 3. 권한 구조(Separation) 코어 아키텍처

앱의 보안 권한 처리는 `ui/main_window.py` 의 **QStackedWidget** 라우팅 로직을 기반으로 합니다.

1. **Routing Switch (게이트웨이)**:
   * 최초 앱 구동 시 `MainWindow`는 어떠한 화면도 먼저 보여주지 않고 `LoginView` (게이트웨이)에 사용자들을 가둡니다.
   * 유저/관리자 버튼의 고유 `Signal(시그널)`에 따라 `show_user_dashboard()` 뷰와 `show_admin_dashboard()` 뷰로 통로가 완벽히 갈라집니다.

2. **비즈니스 분리 (Decoupling)**:
   * 유저의 액션은 오로지 `ScoringEngine`(분석/읽기) 권한 함수만 호출하도록 연결되어 있습니다. (`Read-Only` 성향)
   * 관리자의 액션은 `DatabaseManager`의 `INSERT/UPDATE`와 `FullDataCollector` 클래스(수집/쓰기)만을 호출하도록 설계되었습니다. (`Write/Execute` 성향)

이를 통해 유저의 단순 실수나 해킹으로 애플리케이션의 DB가 비워지거나 API 과도 호출로 인한 계정 차단(Rate Limit 밴)이 발생하는 것을 기술적으로 원천 방어합니다.
